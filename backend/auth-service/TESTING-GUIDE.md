# Complete Testing Guide for Auth Service

This guide provides a comprehensive checklist to manually verify every component of the `auth-service`.

## 1. Prerequisites Verification
Before making API calls, ensure your infrastructure is running.

- **PostgreSQL Database:**
  - Connect using a tool like DBeaver, pgAdmin, or psql.
  - **Host:** `localhost`
  - **Port:** `5432`
  - **Database:** `auth_db`
  - **Username/Password:** `postgres` / `password`
  - *Check:* You should see tables like `user_credentials`, `roles`, `refresh_tokens`, `login_attempts`, etc., generated by Hibernate.

- **Redis Server:**
  - Connect using Redis CLI or RedisInsight.
  - **Host/Port:** `localhost:6379`
  - *Check:* Run `PING` in the CLI; it should return `PONG`.

---

## 2. API Endpoints Testing (using Postman)

### 2.1 Registration (`POST /api/v1/auth/register`)
- **Positive Test:** Send valid `email`, `phone`, `password` (>8 chars), and `role` (e.g., `DRIVER`).
  - *Verify:* You get a `200 OK`. `accessToken` and `refreshToken` are in the response.
- **Negative Test (Duplicate Email):** Try to register with the exact same email again.
  - *Verify:* You get a `400 Bad Request` with message "Email already in use".
- **Negative Test (Validation):** Send a password with only 3 characters.
  - *Verify:* You get a `400 Bad Request` citing a `VALIDATION_ERROR` for the password size.

### 2.2 Login & Brute Force Protection (`POST /api/v1/auth/login`)
- **Positive Test:** Login with the email and password you just registered.
  - *Verify:* `200 OK` with fresh tokens.
- **Negative Test (Bad Password):** Send the wrong password.
  - *Verify:* `401 Unauthorized` with "Invalid credentials".
- **Brute Force Test:** Send the wrong password **5 times in a row**.
  - *Verify:* On the 5th or 6th try, you get a `403 Forbidden` with "Account locked due to too many failed attempts".

### 2.3 Refresh Token (`POST /api/v1/auth/refresh`)
- **Positive Test:** Send the `refreshToken` you received from the login response.
  - *Verify:* `200 OK` with a brand new `accessToken`.
- **Negative Test:** Send a randomly typed fake token.
  - *Verify:* `400 Bad Request` saying "Refresh token not found".

### 2.4 Logout & JWT Blacklisting (`POST /api/v1/auth/logout`)
- **Positive Test:** Send a request with `Authorization: Bearer <your_access_token>`.
  - *Verify:* `200 OK`.
- **Verify Blacklist:** Try to use that exact same `accessToken` on a protected endpoint (if you had one) or try to use the `refreshToken` you just logged out with on the `/refresh` endpoint.
  - *Verify:* The refresh token should now say "Refresh token expired or revoked" because it was flagged in the database.

---

## 3. Database State Verification
After running the Postman tests, check your PostgreSQL database to verify data integrity:

1. **Table `user_credentials`:** Run `SELECT * FROM user_credentials;`
   - Ensure the password is encrypted (looks like `$2a$10$...`).
   - Ensure `enabled` is `true`.
2. **Table `roles`:** Run `SELECT * FROM roles;`
   - You should see the `DRIVER` role (or whatever you used in registration) created automatically.
3. **Table `user_roles`:** Run `SELECT * FROM user_roles;`
   - Verify your user ID is mapped to the role ID.
4. **Table `refresh_tokens`:** Run `SELECT * FROM refresh_tokens;`
   - You should see tokens stored. If you tested logout, the `revoked` column should be `true` for that token.

---

## 4. Redis State Verification
To verify Redis is correctly tracking login attempts and token blacklists:

1. Open your Redis CLI (`redis-cli` or via Docker: `docker exec -it auth-redis redis-cli`).
2. Run `KEYS *`.
3. If you failed a login, you will see a key like `"login_attempt:0:0:0:0:0:0:0:1"` (your local IP).
4. Run `GET "login_attempt:0:0:0:0:0:0:0:1"` to see the counter value.
5. If you logged out, you will see a key like `"blacklist:<your_access_token>"`.

---

## 5. JWT Verification (Security Check)
1. Go to **[jwt.io](https://jwt.io)**.
2. Paste the `accessToken` you got from Postman.
3. **Check the Payload:**
   - `sub`: Should be the registered email address.
   - `iat`: Issued At timestamp.
   - `exp`: Expiration timestamp (Should be exactly 15 minutes after `iat`).

---

## 6. Exception & Logging Verification
Check the terminal where you ran `mvn spring-boot:run`.
- Verify you see logs generated by `EmailService.java` (e.g., `Sending welcome email to testdriver@example.com`).
- Verify SQL queries are being printed (because `show-sql: true` is enabled).
